# Récif API

The Récif API is a Go HTTP server built with [Chi](https://github.com/go-chi/chi). It provides agent management, chat proxying, evaluations, and governance endpoints.

## Base URL

```
http://localhost:8080/api/v1
```

All endpoints under `/api/v1` require authentication when `auth.enabled=true` in the Helm values. Authentication uses JWT tokens validated via the configured OIDC provider.

## Agent endpoints

### Create agent

```
POST /api/v1/agents
```

**Request body:**

```json
{
  "name": "My Agent",
  "framework": "adk",
  "description": "A helpful assistant",
  "version": "1.0.0",
  "config": {}
}
```

- `name` (required): Display name (1-255 characters).
- `framework` (required): One of `adk`, `langchain`, `crewai`, `autogen`.
- `version` (required): Semantic version string (1-20 characters).
- `description` (optional): Up to 500 characters.
- `config` (optional): Arbitrary JSON configuration.

If an agent with the same slug (derived from name) already exists, this performs a **re-registration**: updates the agent and creates a new version entry.

**Response:** `201 Created` (new) or `200 OK` (re-registration)

```json
{
  "data": {
    "id": "ag_01JEXAMPLE00000000000000",
    "team_id": "tk_DEFAULT000000000000000000",
    "name": "My Agent",
    "slug": "my-agent",
    "status": "registered",
    "framework": "adk",
    "version": "1.0.0",
    "created_at": "2026-03-16T10:00:00Z",
    "updated_at": "2026-03-16T10:00:00Z"
  }
}
```

### List agents

```
GET /api/v1/agents
GET /api/v1/agents?search=keyword
```

Returns up to 100 agents. Optional `search` query parameter filters by name.

**Response:** `200 OK`

```json
{
  "data": [
    { "id": "ag_...", "name": "My Agent", "status": "active", ... }
  ]
}
```

### Get agent

```
GET /api/v1/agents/{id}
```

**Response:** `200 OK` or `404 Not Found`

### List agent versions

```
GET /api/v1/agents/{id}/versions
```

Returns all version snapshots for an agent, ordered by creation time.

### Update agent status

```
PATCH /api/v1/agents/{id}/status
```

**Request body:**

```json
{
  "status": "active"
}
```

Valid status transitions follow the agent lifecycle: `registered` -> `draft` -> `active` -> `archived`.

## Chat proxy endpoints

These endpoints proxy requests to the agent's Corail Pod via in-cluster DNS. Use the agent **slug** (not the `ag_` ID).

### Chat (SSE streaming)

```
POST /api/v1/agents/{slug}/chat
```

**Request body:**

```json
{
  "input": "Hello, agent!",
  "conversation_id": "conv-001"
}
```

**Response:** SSE stream with `event: ` and `data: ` fields.

### Chat stream

```
POST /api/v1/agents/{slug}/chat/stream
```

Same as `/chat` but explicitly streaming. The response uses `text/event-stream` content type with chunked transfer.

**SSE event payloads** (all delivered as `data: <json>\n\n` lines):

| Field | When | Meaning |
|-------|------|---------|
| `token` | Every model chunk | One piece of the response text. Thinking tokens are wrapped in `<think>…</think>`. |
| `component` / `confirm` | When a skill emits a structured block | [AG-UI](../corail/agui.md) structured output — renderable component or confirmation prompt. |
| `conversation_id` | First chunk of a new conversation | The server-assigned conversation id. |
| `done` | Once the model stream is complete | Frontend should unlock the input field. The stream may still deliver suggestions after this. |
| `error` | When the pipeline raises | A human-readable error string, surfaced so the user sees what went wrong instead of an empty reply. |
| `suggestions` | After `done`, when background model replies | Array of 2-3 follow-up questions generated by the background model. Bounded by `CORAIL_SUGGESTIONS_TIMEOUT` (15 s by default) so the SSE connection never lingers. |

### Update agent config

```
PATCH /api/v1/agents/{id}/config
```

Updates the Agent CRD spec in Kubernetes. Accepts a partial JSON body with the fields to change:

```json
{
  "modelType": "anthropic",
  "modelId": "claude-sonnet-4-20250514",
  "strategy": "agent-react"
}
```

The operator reconciles the change, updates the ConfigMap, and the config hash annotation triggers a rolling restart of the agent Pod.

### List conversations

```
GET /api/v1/agents/{id}/conversations
```

Proxied to the agent Pod's `/conversations` endpoint.

### Get conversation detail

```
GET /api/v1/agents/{id}/conversations/{cid}
```

Proxied to the agent Pod's `/conversations/{cid}` endpoint.

### Delete conversation

```
DELETE /api/v1/agents/{id}/conversations/{cid}
```

Proxied to the agent Pod. Removes the conversation and all its messages.

### Generation status

```
GET /api/v1/agents/{id}/conversations/{cid}/status
```

Proxied to the agent Pod. Returns whether the agent is currently generating a response for this conversation.

## Evaluation endpoints

### Create dataset

```
POST /api/v1/agents/{agent_id}/datasets
```

### Trigger evaluation

```
POST /api/v1/agents/{agent_id}/evaluations
```

### Eval result callback

```
POST /api/v1/agents/{id}/releases/{version}/eval-result
```

**Request body** (POSTed by Corail after evaluation completes):

```json
{
  "run_id": "eval_abc123",
  "status": "completed",
  "scores": {"safety": 0.95, "correctness": 0.82},
  "passed": true,
  "verdict": "PASSED (avg=0.885 >= 0.750)"
}
```

If `passed=true`, the release is promoted to `active`. If `passed=false`, the release is marked `rejected` and the CRD is rolled back.

### Submit feedback

```
POST /api/v1/feedback
```

```json
{
  "trace_id": "tr-abc123",
  "name": "user_rating",
  "value": 4.0,
  "source": "user",
  "comment": "Good response but too verbose",
  "agent_id": "ag_01J..."
}
```

Proxied to MLflow as a trace assessment. Negative feedback (value < 3) is flagged for eval dataset inclusion.

### Flagger quality gate webhook

```
POST /api/v1/webhooks/flagger
```

Called by Flagger during canary analysis. Returns `{"status": "ok"}` if the quality gate passes.

## Knowledge base endpoints

### List knowledge bases

```
GET /api/v1/knowledge-bases
```

**Response:** `200 OK`

```json
{
  "data": [
    {
      "id": "kb_01J...",
      "team_id": "tk_DEFAULT...",
      "name": "Product Docs",
      "slug": "product-docs",
      "description": "Product documentation",
      "embedding_model": "nomic-embed-text",
      "embedding_dim": 768,
      "chunk_size": 500,
      "chunk_overlap": 50,
      "doc_count": 12,
      "chunk_count": 347,
      "status": "ready",
      "created_at": "2026-03-18T10:00:00Z"
    }
  ]
}
```

### Create knowledge base

```
POST /api/v1/knowledge-bases
```

**Request body:**

```json
{
  "name": "Product Docs",
  "description": "Product documentation for RAG",
  "embedding_model": "nomic-embed-text"
}
```

Chunk size and overlap are not exposed here — the server uses sensible defaults (512 / 50). The `embedding_model` is the one the retriever will use; it is passed through to Marée on every ingestion so the stored vectors always match the retrieval path.

**Response:** `201 Created`

### Get knowledge base

```
GET /api/v1/knowledge-bases/{id}
```

### Delete knowledge base

```
DELETE /api/v1/knowledge-bases/{id}
```

Removes the KB, all its documents and chunks via `ON DELETE CASCADE`, and best-effort deletes the upload directory on disk. Returns `204 No Content` on success, `404 Not Found` if the KB does not exist.

### List documents

```
GET /api/v1/knowledge-bases/{id}/documents
```

Returns all documents ingested into the KB. Each row includes `status` (`pending` → `processing` → `ready` or `error`), `chunk_count`, and — when ingestion failed — a truncated `error` field with the Marée stderr so the dashboard can surface it without tailing pod logs.

### Delete document

```
DELETE /api/v1/knowledge-bases/{id}/documents/{docId}
```

Removes a single document, its chunks (via the same cascade), and best-effort deletes the uploaded file from disk. Returns `204 No Content`.

### Ingest document

```
POST /api/v1/knowledge-bases/{id}/ingest
Content-Type: multipart/form-data
```

Upload a file for ingestion. The API:

1. Saves the file at `/tmp/maree-uploads/{kbId}/{filename}` inside the `recif-api` pod
2. Creates a `kb_documents` row with `status='pending'` and returns it to the caller immediately
3. Flips the row to `status='processing'` and spawns `maree ingest` as a background subprocess with `--document-id` set to the row's id so Marée populates the existing row instead of creating a parallel one
4. Captures the subprocess stderr — if it exits non-zero, the row is marked `status='error'` with a truncated error message

Marée's CLI lives in the same container image and uses the Knowledge Base's `embedding_model` to generate vectors. Chunks land in the sibling `corail_storage` database (not the main `recif` one) where the agent retriever reads them.

**Form field:** `file` (the document to ingest — .txt, .md, .pdf, .docx)

**Response:** `201 Created`

```json
{
  "data": {
    "id": "doc_01J...",
    "kb_id": "kb_01J...",
    "filename": "guide.pdf",
    "status": "pending",
    "chunk_count": 0
  },
  "meta": {
    "upload_path": "/tmp/maree-uploads/kb_01J.../guide.pdf",
    "message": "File uploaded. Ingestion started."
  }
}
```

The dashboard polls `GET /knowledge-bases/{id}/documents` every 2 s while any row is in `pending` or `processing` so the status flips to `ready` (with the final `chunk_count`) or `error` (with the reason) without the user having to refresh.

### Search knowledge base

```
POST /api/v1/knowledge-bases/{id}/search
```

Performs vector similarity search. Embeds the query via Ollama, then searches pgvector.

**Request body:**

```json
{
  "query": "How do I deploy an agent?",
  "top_k": 5
}
```

**Response:** `200 OK`

```json
{
  "data": [
    {
      "chunk_id": "abc123...",
      "content": "To deploy an agent, create an Agent CRD...",
      "score": 0.87,
      "chunk_index": 3,
      "document_id": "doc_...",
      "filename": "guide.pdf"
    }
  ]
}
```

## Error format

All errors follow a consistent JSON structure:

```json
{
  "error": "Not Found",
  "message": "Agent not found",
  "path": "/api/v1/agents/nonexistent"
}
```
